# How to Emulate LEFT JOIN FETCH Using Record-Based Projections

## Summary
How to emulate a LEFT JOIN FETCH between two root entities without fetching the intermediate child entity in Hibernate. Uses record-based projections to achieve efficient query patterns that avoid the N+1 problem while keeping the object graph clean.

## Why This Matters
LEFT JOIN FETCH is the standard Hibernate solution for the N+1 problem, but it has limitations when dealing with intermediate entities. Record-based projections (Java 16+) provide a cleaner alternative for specific query patterns.

## Key Points
- LEFT JOIN FETCH loads associated entities in a single query
- Intermediate entities can cause unnecessary data loading
- Record-based projections allow selective field fetching
- Works with Spring Data JPA and Hibernate native queries

## How to Use Record-Based Projections

### Define the Projection Record
```java
public record EmployeeDto(
    Long id,
    String name,
    String departmentName,
    String departmentLocation
) {}
```

### Create the Repository Method
```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<EmployeeDto> findAllWithDepartmentAsDto();
}
```

### Generated Query (Spring Data JPA Derives the Query)
```sql
SELECT e.id, e.name, d.name, d.location
FROM Employee e
LEFT JOIN Department d ON e.department_id = d.id
```

## Common Pitfalls
- **Projection mutability**: Records are immutable — you can't modify projected results
- **Missing null handling**: LEFT JOIN returns NULL for unmatched rows — handle nulls in record constructors
- **Performance**: Projections bypass Hibernate's first-level cache — consider caching strategies for frequently accessed data
